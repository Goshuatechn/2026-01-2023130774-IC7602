## 1. Explique el funcionamiento de ICMP

El Internet Control Message Protocol (ICMP) es un protocolo que existe dentro del ecosistema de IP con un propósito muy específico: proveer retroalimentación sobre problemas en el entorno de comunicación. Es importante entender desde el principio que IP no fue diseñado para ser absolutamente confiable, entonces ICMP no busca hacer que IP sea confiable, sino simplemente reportar lo que sale mal cuando algo sale mal. Igual pueden existir datagramas que se pierdan sin que nadie lo reporte, ya que esa responsabilidad de garantizar confiabilidad recae en los protocolos de capas superiores.

ICMP usa el soporte básico de IP como si fuera un protocolo de capa superior, aunque en la práctica es una parte integral de IP y todo módulo IP debe implementarlo. Los mensajes ICMP viajan encapsulados en datagramas IP usando el encabezado estándar, donde el campo de protocolo toma el valor 1 para identificar que el payload es un mensaje ICMP. Cada mensaje ICMP tiene un campo de tipo que determina el formato del resto del mensaje, un código que especifica la razón dentro de ese tipo, y un checksum para verificar la integridad.

Para evitar que los mensajes de error generen más mensajes de error en cadena infinita, ICMP tiene una regla clara: no se envían mensajes ICMP en respuesta a otros mensajes ICMP. Además, los mensajes ICMP solo se generan para errores en el fragmento cero de datagramas fragmentados, ya que es ese el que tiene el fragment offset igual a cero.

Los tipos de mensajes más relevantes que define el protocolo son los siguientes.

El **Destination Unreachable** (tipo 3) es el que se genera cuando un gateway consulta su tabla de enrutamiento y determina que la red destino es inalcanzable, o cuando el host destino no puede entregar el datagrama porque el protocolo o el puerto indicado no está activo. También se usa cuando un datagrama necesita ser fragmentado pero tiene el flag Don't Fragment activado, en cuyo caso el gateway descarta el paquete y puede devolver este mensaje. Los códigos van del 0 al 5 y cubren casos como red inalcanzable, host inalcanzable, protocolo inalcanzable, puerto inalcanzable, fragmentación necesaria con DF activo y ruta origen fallida.

El **Time Exceeded** (tipo 11) se genera en dos situaciones. La primera es cuando el campo Time to Live de un datagrama llega a cero en tránsito por un gateway, lo que hace que ese gateway descarte el paquete y pueda notificar al origen con código 0. La segunda es cuando un host está intentando rearmar un datagrama fragmentado y no lo logra dentro del tiempo límite porque le faltan fragmentos, en cuyo caso descarta lo que tiene y puede enviar este mensaje con código 1. Si el fragmento cero no está disponible, no se envía ningún mensaje.

El **Parameter Problem** (tipo 12) ocurre cuando un gateway o host encuentra un problema con los parámetros del encabezado del datagrama que le impide procesarlo correctamente, como argumentos incorrectos en una opción. En este caso el datagrama se descarta y se puede notificar al origen. El mensaje incluye un campo Pointer que identifica el octeto exacto del encabezado original donde se detectó el error.

El **Source Quench** (tipo 4) es un mecanismo de control de flujo. Cuando un gateway no tiene suficiente espacio en su buffer para encolar datagramas que va a reenviar, puede descartar datagramas y enviar un Source Quench al origen por cada uno que descarte. Un host destino también puede enviarlo si los datagramas llegan demasiado rápido para procesarlos. Al recibir este mensaje, el origen debe reducir la tasa a la que envía tráfico hacia ese destino, y puede ir aumentándola gradualmente hasta que vuelva a recibir otro Source Quench. Lo interesante es que el gateway puede enviar este mensaje incluso antes de que su capacidad sea excedida, como medida preventiva.

El **Redirect** (tipo 5) se usa cuando un gateway detecta que hay una ruta más corta para llegar al destino. Por ejemplo, si el gateway G1 recibe un datagrama de un host y al consultar su tabla de enrutamiento encuentra que el siguiente salto es G2, y tanto G2 como el host origen están en la misma red, entonces G1 le envía un Redirect al host indicándole que mande ese tráfico directamente a G2. El gateway igual reenvía el datagrama original a su destino. Los códigos permiten redirigir por red, por host, por tipo de servicio y red, o por tipo de servicio y host.

El **Timestamp y Timestamp Reply** (tipos 13 y 14) permiten medir la latencia entre dispositivos. El mensaje incluye tres timestamps de 32 bits en milisegundos desde medianoche UT: el Originate Timestamp que marca cuándo el emisor envió el mensaje, el Receive Timestamp que marca cuándo el receptor lo tocó por primera vez, y el Transmit Timestamp que marca cuándo el receptor lo envió de vuelta. Con estos tres valores se puede calcular el tiempo de tránsito en cada dirección.

El **Information Request y Reply** (tipos 15 y 16) permiten que un host descubra el número de red en la que se encuentra, enviando un mensaje con las direcciones origen y destino en cero, y el receptor responde con las direcciones completamente especificadas.

## 2. Aplicaciones de ICMP en las comunicaciones

Aunque ICMP es un protocolo de control y no de transporte de datos, su utilidad práctica en las comunicaciones es enorme ya que es la base de varias herramientas y mecanismos que se usan constantemente en la administración y diagnóstico de redes.

La herramienta más conocida que se apoya en ICMP es **ping**, que usa los mensajes Echo Request y Echo Reply para verificar si un host remoto está activo y alcanzable, y aparte de eso mide el tiempo de ida y vuelta (RTT) entre ambos puntos. Esto lo hace directamente desde los Timestamp o desde el mecanismo de echo que el protocolo define, y es el primer recurso que se usa cuando se sospecha que hay un problema de conectividad.

Otra herramienta fundamental es **traceroute**, que explota el mecanismo de Time Exceeded de ICMP para mapear la ruta completa que sigue un paquete desde el origen hasta el destino. Lo que hace es enviar datagramas con TTL incremental empezando desde 1, entonces cada gateway que descarta el paquete por TTL agotado devuelve un Time Exceeded, lo que le permite a traceroute ir conociendo la dirección IP de cada salto en la ruta y el tiempo que tarda en llegar a cada uno.

El mecanismo de **Source Quench** tiene una aplicación directa en el control de congestión de la red, ya que permite que los gateways indiquen a los orígenes que reduzcan su tasa de envío antes de que el problema se vuelva crítico, funcionando como un freno de emergencia distribuido en la red.

Los mensajes de **Destination Unreachable** tienen aplicación en el diagnóstico de problemas de conectividad más específicos, ya que los distintos códigos permiten distinguir si el problema es que la red no existe en las tablas de enrutamiento, que el host específico no responde, que el protocolo no está soportado en el destino o que el puerto al que se intenta conectar no tiene ningún proceso escuchando. Esta granularidad es lo que hace que las aplicaciones de red puedan dar mensajes de error descriptivos al usuario en lugar de simplemente decir que algo falló.

El **Redirect** tiene aplicación directa en la optimización dinámica del enrutamiento a nivel de host, ya que permite que los gateways corrijan las rutas subóptimas que un host está usando sin necesidad de intervención manual, haciendo que el tráfico tome caminos más eficientes de forma automática conforme la red cambia.

Finalmente, los mensajes de **Timestamp** tienen aplicación en la sincronización de relojes entre dispositivos de red y en la medición de latencia asimétrica entre dos puntos, lo que es relevante para aplicaciones sensibles al delay como videoconferencia o VoIP, donde saber cuánto tarda un paquete en cada dirección permite ajustar los buffers y estrategias de compensación de jitter.

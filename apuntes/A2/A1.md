## 1. DNS — Domain Name Service

El DNS es basicamente una base de datos NoSQL de clave-valor, donde su funcion es asociar un nombre de dominio con una direccion IP. Nereo usa una analogía bastante ilustradora, "dentro de tu casa es "Nereo", pero cuando hay que salir al mundo hay que presentarse como "Nereo Campos Araya" con tu cedula". De la misma manera, un nombre de maquina como `www` es solo una parte de la identidad, mientras que `www.google.com` es el nombre completo y único de Internet, esto es conocido como **FQDN (Fully Qualified Domain Name)**.

---

## 2. El Resolver

Cualquier dispositivo conectado a una red tiene un proceso llamado **resolver**, cuya función es hacer preguntas al DNS. Este proceso lee un archivo de configuración llamado `resolv.conf` para saber a qué servidor DNS debe consultar, y mantiene una caché local para no tener que preguntar lo mismo dos veces.

El orden de resolución que sigue el resolver es el siguiente:

1. Primero consulta el archivo **hosts** (uno de los archivos más peligrosos del sistema, porque si alguien lo modifica puede redirigir cualquier dominio a un servidor diferente sin que el usuario se dé cuenta).
2. Luego revisa su **caché** local.
3. Si no encuentra la respuesta, hace una consulta al **servidor DNS** configurado.

El **TTL (Time to Live)** es el parámetro que define cuánto tiempo se mantiene una entrada en la caché del resolver. No significa que la IP cambia cada cierto tiempo, sino que después de ese tiempo el resolver debe volver a preguntar para asegurarse de que la información sigue siendo válida.

## 3. Características del Protocolo DNS

El DNS opera sobre **UDP puerto 53** — un protocolo no orientado a conexión, lo que significa que simplemente se manda la petición y se espera respuesta sin establecer una conexión formal. Además, el protocolo es **binario, no ASCII**, lo que quiere decir que no se puede leer directamente — se necesita un editor hexadecimal para interpretar lo que está pasando.

Es importante saber que el DNS es uno de los protocolos más antiguos de Internet (tiene alrededor de 44 años) y prácticamente no ha cambiado en su núcleo. Lo que sí ha evolucionado es toda la infraestructura que empresas como Google, Microsoft y Amazon han construido alrededor de él.

El DNS maneja distintos **tipos de registros**, entre los más relevantes:

| Tipo | Descripción |
|---|---|
| **A** | Asocia un nombre de dominio a una dirección IPv4 |
| **AAAA** | Asocia un nombre de dominio a una dirección IPv6 |
| **CNAME** | Alias — resuelve un nombre de dominio a otro nombre de dominio |
| **MX** | Registros de correo electrónico (mail exchange) |

También existen dos tipos de consultas: la **estándar** (preguntás un nombre y te devuelve una IP) y la **inversa** (preguntás una IP y te devuelve el nombre de dominio asociado).

## 4. DNS Interceptor y DNS Forwarder

El **DNS Interceptor** es un componente que se ejecuta localmente en una computadora y levanta un socket en el puerto UDP-53. Al configurar el archivo `resolv.conf` para que apunte a `127.0.0.1:53`, todas las peticiones DNS del sistema pasan primero por el interceptor antes de salir a Internet. Funciona esencialmente como un proxy de DNS.

El **DNS Forwarder** es un servidor externo al que el interceptor reenvía las peticiones que no puede resolver por sí mismo. Puede apuntar a servidores conocidos como `8.8.8.8` de Google o `9.9.9.9` de Cloudflare.

Este esquema permite interceptar y analizar todo el tráfico DNS de una máquina, modificar respuestas, aplicar reglas de enrutamiento geográfico, y mucho más.

---

## 5. Geolocalización y Round Robin

Los bloques de IPs públicas se asignan geográficamente, lo que significa que a partir de una IP se puede determinar el país o región de origen. Usando una base de datos de **IP to Country**, es posible detectar desde dónde viene una petición DNS y responder con la IP más conveniente según esa ubicación.

El **Round Robin** es un algoritmo simple que rota entre varias IPs disponibles (IP1, IP2, IP3...) para distribuir la carga de forma equitativa. Combinado con **Geo Location**, se puede ser más inteligente: en lugar de simplemente rotar, se elige la IP más cercana al cliente según su ubicación geográfica y la latencia medida.

## 6. Health Checker

El Health Checker es un componente que monitorea continuamente la disponibilidad de los servidores haciendo peticiones periódicas y registrando si responden o no, y en cuánto tiempo. Algunas consideraciones importantes:

- No se recomienda hacer más de **30 peticiones por minuto** por servidor, o el sistema puede detectar el tráfico como un bot.
- Si se tienen múltiples Health Checkers distribuidos geográficamente (por ejemplo uno en Cartago y otro en Alajuela), se puede usar la latencia medida desde cada punto para determinar cuál servidor está más cerca del cliente que hace la petición.
- Si hay varios Health Checkers y la mayoría reporta que un servidor está disponible, se toma como que está bien (por ejemplo, si 11 de 20 dicen que está arriba, se asume disponible).
- Cuando un servidor no responde, el Health Checker actualiza la base de datos para que el DNS Interceptor deje de enviarle tráfico, garantizando **alta disponibilidad**.

## 7. Consideraciones de Implementación

Algunos problemas conocidos al implementar un DNS Interceptor:

- Los sistemas operativos modernos con antivirus **protegen activamente el DNS** y pueden bloquear cualquier intento de modificarlo. Probablemente sea necesario desactivar el antivirus durante las pruebas.
- En **Linux y Mac**, el sistema ya tiene su propio DNS Interceptor interno para análisis de tráfico, por lo que hay que apagarlo explícitamente.
- **Kubernetes** tiene restricciones para usar el puerto 53-UDP, lo que puede generar muchos problemas. La recomendación es no correrlo en Kubernetes sino en una máquina virtual en el cloud.

La prueba de fuego del proyecto es levantar el interceptor y verificar que el tráfico DNS pase correctamente a través de él. El código debe ser lo más eficiente posible porque cualquier lentitud se acumula en forma de timeouts en cadena.

## 8. Detalles del Proyecto

El proyecto consiste en construir un sistema DNS propio con capacidades de interceptación, geolocalización y alta disponibilidad. Los componentes principales mencionados en clase son:

**DNS Interceptor**
Es el núcleo del proyecto. Se implementa en C++ y escucha en el puerto UDP-53. Intercepta todas las peticiones DNS del sistema, las decodifica (el paquete llega en base64), consulta la base de datos para determinar qué IP responder según la ubicación del cliente, y construye el paquete de respuesta DNS en formato binario. Debe soportar múltiples solicitudes simultáneas para no convertirse en un cuello de botella.

**Base de datos (Supabase)**
Se usará Supabase (basada en PostgreSQL) en lugar de Firebase, ya que Firebase tiene limitantes en la cantidad de registros que se pueden consultar. La base de datos almacena los dominios configurados, las IPs asociadas, la tabla de IP to Country (que debe cargarse como CSV), y las métricas de latencia reportadas por los Health Checkers.

**IP to Country**
Se descarga una base de datos pública que asocia rangos de IPs con países y regiones. Esto permite que el interceptor detecte desde dónde viene una petición y tome decisiones de enrutamiento geográfico.

**Health Checker**
Corre en múltiples ubicaciones geográficas (al menos dos). Hace peticiones periódicas a los servidores registrados, mide la latencia y actualiza la base de datos con el estado de cada IP. El límite recomendado es no superar 30 peticiones por minuto para no ser detectado como bot. Si la mayoría de los checkers reporta que un servidor está caído, se marca como no disponible y el interceptor deja de enviarle tráfico.

**DNS UI**
Interfaz en React que permite hacer CRUD sobre los dominios e IPs configurados en la base de datos.

**Despliegue**
Se recomienda correrlo en una máquina virtual en el cloud en lugar de Kubernetes, dado que este último tiene restricciones con el puerto 53-UDP. La referencia técnica del protocolo DNS es el **RFC1035**.
